# AIO Constitution

## Core Principles

### Article I — Python 3.12+ as the Platform Language

AIO MUST be developed targeting Python 3.12+ as the primary runtime. No backward
compatibility with Python < 3.10 is required.

- Use walrus `:=`, type unions `A | B`, `match/case`, `functools.cache`, `tomllib` (built-in), and modern async/await.
- MUST NOT use `typing.Union[A, B]` — use `A | B` directly.
- Type hints are first-class: `@dataclass`, `TypedDict`, `Protocol` used freely.
- `from __future__ import annotations` is allowed EXCEPT in `cli.py` and `serve.py`, which rely on Typer's runtime type introspection.
- New modules MUST run on 3.10+ for extended support where applicable, but MUST NOT constrain features for backward compat.

### Article II — 100% Standalone Offline HTML Output (Portability Law)

Every presentation generated by AIO MUST be a single self-contained `.html` file
with zero runtime external dependencies.

- All assets (CSS, JS, SVG, woff2 fonts, base64 images) MUST be inlined inside `<style>` or `<script>` tags.
- Offline-first is NON-NEGOTIABLE: a presentation opened on any machine (airplane, no internet) MUST work 100%.
- Reveal.js vendor code is static; upgrading it is a manual step, never a runtime dependency.
- Images generated via `--enrich` MUST be downloaded at build time and embedded as base64 or inline SVG.
- No `<link href="https://...">` and no `<script src="https://...">` are allowed in output HTML.
- Build Step 5 (`inline_assets`) MUST fail with exit code 3 if any external URL is detected in output.

### Article III — Visual Intelligence as Core Differentiator

Visual composition, intelligent layout inference, SVG composites, and data
visualization are NOT optional features — they ARE the product. Every architectural
decision MUST prioritize visual expressiveness.

- `src/aio/composition/` (Composition Engine) is the architectural foundation, not an add-on.
- 16+ layouts MUST be planned from M0 onward, not retrofitted.
- SVG composites (`src/aio/visuals/svg/`) MUST render complex static illustrations.
- DataViz Engine (`src/aio/visuals/dataviz/`) MUST convert CSV/JSON/inline data to SVG charts
  (bar, line, pie, scatter, heatmap).
- Icon Library (~200 Lucide icons) is vendored; MUST support CSS customization.
- Every slide MUST pass through `CompositionEngine.infer_layout()` — no slide exits the renderer
  without layout inference applied.
- Tests MUST include SVG well-formedness validation.

### Article IV — Agent-Agnostic Multimodal Prompting

AIO MUST natively support 8 AI agents, each with its own versioned prompt template.
No agent is second-class. Intelligence lives in the prompts, not the orchestrator.

Supported agents: Claude (Anthropic, default), Gemini (Google), Copilot (Microsoft),
Windsurf (Codeium), Devin (Cognition), ChatGPT (OpenAI), Cursor (Anysphere), Generic REST API.

- `src/aio/agent_commands/` MUST contain 8 subdirectories, each with:
  `SYSTEM.md`, `INIT_PHASE.md`, `COMPOSITION_PHASE.md`, `ENRICH_PHASE.md`, `REFINEMENT_PHASE.md`.
- Prompts MUST be vendorized at each release (frozen at build time — never live-fetched).
- Multi-modal inputs (text + image for visual context) MUST be supported.
- CLI `--agent` flag selects the template; default is `claude`.
- Each agent MUST pass a smoke test: generate a 5-slide presentation validating HTML structure
  and visual composition.

### Article V — DESIGN.md: Dual-Purpose Human + Machine Spec

Every theme MUST have a `DESIGN.md` that simultaneously serves as human documentation
AND as a machine-parseable specification for `extract.py` and prompt templates.

The following 11 sections are MANDATORY in every `DESIGN.md`:
1. Visual Theme
2. Color Palette (hex values + roles + contrast ratios)
3. Typography (fonts, sizes, line-heights, weights)
4. Components (buttons, badges, cards, callouts, code blocks)
5. Layout System (grid, spacing scale, breakpoints, safe areas)
6. Depth & Shadows (elevation 0–4, shadow definitions)
7. Do's & Don'ts (required patterns + anti-patterns)
8. Responsive Behavior (mobile-first, tablet/desktop overrides)
9. Animation & Transitions (speeds, easing, triggers)
10. Accessibility (WCAG 2.1 AA claims, contrast guarantees, alt-text rules)
11. Agent Prompt Snippet (~200 words, executable by any supported agent)

- `extract.py` performs structured parsing; breaking the format MUST be treated as a breaking change.
- No format changes are allowed without a formal Amendment and simultaneous update of `extract.py`.
- `aio theme validate {id}` MUST verify DESIGN.md well-formedness and parser compatibility.

### Article VI — Free by Default (Economic Accessibility Law)

Every "premium" feature (image generation, visual richness, enrichment) MUST work
for free by default via Pollinations.ai (open API, no API key required).

- `--enrich` MUST use Pollinations.ai as the default provider — no API key required out of the box.
- Paid providers (DALL-E, Midjourney, Stable Diffusion) are OPTIONAL via manual config.
- No core feature MUST require a subscription.
- CLI MUST NOT upsell; if the user wants Pollinations, it works; alternatives are opt-in.
- Logs MUST indicate the active provider; fallback to SVG placeholders if all providers fail.

### Article VII — Minimal but Modern Dependency Stack

AIO MUST use exactly 10 core dependencies (excluding `[enrich]`), all actively
maintained, all < 2 MB each.

**Core (always):** typer >= 0.12, jinja2 >= 3.1, mistune >= 3.0, pyyaml >= 6.0,
rich >= 13.0, click >= 8.1, pygments >= 2.15, watchdog >= 3.0, starlette, uvicorn.

**Optional `[enrich]`:** pillow >= 10.0, beautifulsoup4 >= 4.12, lxml >= 4.9,
cssutils >= 2.7.

**Forbidden (no exceptions):** Django, numpy, pandas, tensorflow (too heavy);
any web framework beyond minimal ASGI; any ORM; GraphQL.

- `pip install -e .` MUST create an environment < 150 MB.
- `pip install -e ".[enrich]"` MUST remain < 250 MB.
- `cssutils` import MUST fail gracefully — never a hard dependency outside `[enrich]`.

### Article VIII — Reveal.js Pinned to 5.x (No Upgrade to 6.x)

Reveal.js MUST remain pinned to `5.0.0 <= version < 6.0`. Upgrading to 6.x is
PROHIBITED without a formal Amendment (breaking API surface, requires full regression).

- Vendor code lives in `src/aio/vendor/revealjs/dist/` as static files — not via npm.
- UMD builds ONLY (required for safe inlining).
- New vendor files MUST be checked for bare `import`/`export` before vendorizing.
- All `</script>` occurrences in inlined JS MUST be escaped to `<\/script>`.
- Build MUST fail if Reveal.js v6+ or bare ES modules are detected.

### Article IX — Test-First, Coverage Grows Toward 80% (TDD Law)

Every new feature MUST have tests written BEFORE implementation (TDD).
The CI coverage gate is ≥ 20% (current baseline); aspirational target is 80% line /
75% branch — coverage MUST only grow, never regress.

**Test structure:**
- `tests/unit/` — isolated functions, mocks allowed.
- `tests/integration/` — full pipeline with real temp directories; NO mocks for core pipeline
  (init, build, compose, enrich).
- `tests/visual/` — HTML snapshots, SVG well-formedness.
- `tests/fixtures/` — shared sample slides, themes, expected outputs, mock API responses.

- `unittest.mock` is FORBIDDEN for the core pipeline — use real temp directories.
- Each CLI command MUST have a smoke test.
- Agent commands MUST be tested with mocks (no real API calls in CI).

### Article X — Theme Library of 64+ Themes (Visual Diversity Law)

AIO MUST launch with 64+ themes from `awesome-design-md`, each 100% functional,
tested, and validated against the DESIGN.md schema.

- Global registry: `src/aio/themes/registry.json` (complete metadata).
- Per-project registry: `.aio/themes/registry.json` (selected theme only — MUST NOT copy the full global registry).
- Each theme MUST have its own directory: `src/aio/themes/{id}/` containing
  `DESIGN.md`, `theme.css`, `layout.css`, `meta.json`.
- `scripts/import-awesome-designs.py` MUST sync nightly via GitHub Actions.
- The fallback theme ("minimal") MUST always be available, even offline.

### Article XI — Agent Commands Versioned and Frozen at Release

Agent prompts (system, init, compose, enrich, refine) MUST be versioned and
frozen at each release. They are NEVER live-fetched or updated via API.

- Each release MUST snapshot `src/aio/agent_commands/{agent}/v{N}/`
  (e.g., `claude/v2/INIT_PHASE.md`).
- Future prompt changes require a CHANGELOG entry + version bump (v2 → v3).
- CLI `--agent-version` MUST support loading older frozen versions.
- Prompts MUST be included in zipapp/PyInstaller builds; no external dependencies.
- Build MUST verify all agent command files are present.

### Article XII — Four Distribution Modes (Ubiquity Law)

AIO MUST function identically in 4 distribution modes; none is "limited" or "beta".

1. **Zero-Install** — `python -m aio --help` (copy `src/aio/` to any directory).
2. **Zipapp** (`.pyz`) — `python aio.pyz --help`.
3. **PyInstaller** (`.exe` / `.bin`) — `./aio --help`.
4. **Pip** — `pip install aio && aio --help`.

- All imports MUST be absolute (no relative imports) to support all 4 modes.
- `sys.path` hacking is FORBIDDEN.
- CI/CD MUST test all 4 modes on Windows, macOS, and Linux before any release.

## Technical Constraints

### Logging & Observability

- Stderr: structured logs (level, component, message) — use `src/aio/_log.py` exclusively.
- Stdout: RESERVED for piping (`aio build | jq`).
- `print()` is FORBIDDEN in production code.
- Log levels: DEBUG, INFO, WARNING, ERROR, CRITICAL.
- Optional log file: `~/.aio/logs/aio.log` (weekly rotation).

### Performance Targets

| Operation | Target |
|-----------|--------|
| Build — 30 slides | < 30s (i5, 8 GB RAM) |
| Serve hot-reload | < 2s |
| Markdown parse — 100 slides | < 500ms |
| New code regression | MUST NOT exceed baseline by > 5% |

### Security

- `yaml.safe_load()` ONLY — `yaml.load()` is FORBIDDEN (prevents RCE via frontmatter).
- SVG sanitization is MANDATORY — no `<script>` tags in any SVG output.
- URLs MUST be validated before any HTTP request.
- HTML output MUST be escaped to prevent XSS when slides contain user-controlled input.

### Documentation

- Docstrings MUST be present in core modules: composition, dataviz, enrich.
- `README.md`, `CONTRIBUTING.md`, and `docs/` (architecture, theme-system, agent-prompts) MUST exist.
- Functional examples MUST exist in `examples/` (5-slide, 50-slide, rich-visuals samples).

## Governance

This constitution supersedes all other technical guidelines and practices. Compliance
is verified in every PR via the "Constitution Check" gate in the implementation plan template.

### Amendment Process

1. **Proposal** — Open a GitHub issue tagged `[constitution]`, arguing why an Article is inadequate.
2. **Discussion** — 48-hour public comment period.
3. **Vote** — Maintainers vote; a 2/3 supermajority is required to approve.
4. **Merge** — Amendment merged, versioned in CHANGELOG, reflected in the next release.
5. **Effective** — New constitution takes effect on the next version tag (`v0.x.0`, etc.).

### Immutable Articles (require hard fork to change)

- **Article I** (Python 3.12+)
- **Article II** (Offline HTML)
- **Article III** (Visual Intelligence as core)
- **Article VIII** (Reveal.js 5.x)

### Versioning Policy

- **MAJOR**: Backward-incompatible governance/principle removal or redefinition.
- **MINOR**: New principle or section added, or materially expanded guidance.
- **PATCH**: Clarifications, wording, typo fixes, non-semantic refinements.
